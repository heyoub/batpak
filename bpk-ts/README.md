# BatPAK TypeScript SDK

Phase 0 — NETBAT/1 wire-parity slice.

This is a **standalone TypeScript SDK** for the BatPAK family. It is not a
product, agent framework, or platform. The single risk it exists to manage
is split-brain: two implementations encoding the same payload to different
bytes, or two schemas describing the same event with subtle skew.

## Phase 0 authority direction

```
Rust descriptor metadata is the source of truth.
  ↓
cargo xtask export-ts-manifest writes ./batpak.manifest.json.
  ↓
@batpak/codegen consumes the manifest, emits @batpak/generated.
  ↓
Parity tests in @batpak/test prove byte equality vs the Rust goldens.
```

`@batpak/schema` exists as a stub for the **future** authoring surface
(Effect Schema v4 beta, pinned exact). It exports `bank.event()` so
downstream callers can pin against the symbol, but calling it in Phase 0
throws — there is no Phase 0 use case for TS-authored event schemas.

## Workspace layout

```
packages/
  canonical/   Named-field MessagePack codec + hex helpers. Byte-parity
               proven against rmp-serde 1.3.1.
  client/      NETBAT/1 frame writer/reader; typed NetbatError union;
               TCP transport.
  codegen/     Reads batpak.manifest.json, emits @batpak/generated.
               Refuses unsupported manifestVersion. Fully overwrites the
               output directory.
  generated/   Auto-generated by codegen. DO NOT HAND-EDIT.
  schema/      Phase 0 stub. Effect Schema v4 beta dep + bank.event()
               placeholder. NOT authoritative in Phase 0.
  test/        Parity harness: payload encode/decode, frame parse/emit,
               manifestVersion guard, ERR UTF-8 guard, denial-vocab
               guard.
examples/
  heartbeat-spike/  Live spike that sends a system.heartbeat to a
                    running hbat and prints the decoded ack.
```

## hbat — the reference host

`hbat` (in `bpk-lib/crates/hbat/`) is the reference NETBAT/1 host. It is
**not a product daemon.** Loopback-only by default; non-loopback binds
require `--allow-non-loopback`. It emits a structured ready line
after binding the TCP listener and before entering the serve loop:

```
HBAT_READY {"addr":"127.0.0.1:54321","port":54321,"protocol":"NETBAT/1"}
```

CI and example scripts parse this exact prefix. No log-output scraping.

## Running locally

```sh
# From the repo root:
cd bpk-lib
cargo run -p xtask -- export-ts-manifest --out ../bpk-ts/batpak.manifest.json

cd ../bpk-ts
pnpm install
pnpm -w build
pnpm -w test

# Live roundtrip:
cd ../bpk-lib
cargo run -p hbat -- serve --store "$(mktemp -d)" --tcp 127.0.0.1:0 --print-port \
  > /tmp/hbat-ready.txt 2>&1 &
HBAT_PID=$!
# Wait until HBAT_READY appears in /tmp/hbat-ready.txt, then:
PORT=$(node -e 'const j=require("fs").readFileSync("/tmp/hbat-ready.txt","utf-8").trim();process.stdout.write(String(JSON.parse(j.replace(/^HBAT_READY /,"")).port))')

cd ../bpk-ts
node examples/heartbeat-spike/dist/index.js --port "$PORT"

kill $HBAT_PID
```

## Determinism

`@batpak/generated/src` is **fully overwritten** on every `pnpm generate`.
Codegen never patches an existing tree. A separate CI step verifies
that running `rm -rf packages/generated/src && pnpm -w build` produces a
byte-identical tree.

## Out of scope (Phase 0)

- NETBAT/2 STREAM (reserved per ADR-0030)
- Effect Schema v4 authoring as source-of-truth (Phase 1+)
- `bank.commit`, `doc.*`, `nav.*`, `receipt.*`, `checkout.*` verbs
- Browser transport, WebSocket bridge, NAPI/WASM acceleration
- A `Bank` Rust type
- camelCase / snake_case field-name transforms on the canonical path
